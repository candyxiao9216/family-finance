## ADR-001：技术栈选择 - Python + SQLite + OpenClaw
- 日期：2026-02-25
- 状态：Accepted（已采纳）
- 背景（要解决的问题）：
  - 需要快速开发全自动 Twitter 运营系统，平衡开发效率、成本和功能需求
- 决策：
  - 使用 Python 3.8+ 作为主要开发语言
  - 使用 SQLite 作为本地数据存储
  - 使用 OpenClaw 进行浏览器自动化控制
- 原因（为什么）：
  - Python 生态系统丰富，AI 集成和数据处理能力强
  - SQLite 轻量级，适合单机部署，无需额外数据库服务
  - OpenClaw 提供稳定的浏览器控制能力，避免反爬虫检测
  - 技术栈组合简单，部署和维护成本低
- 影响（后果）：
  - 正向：
    - 开发速度快，社区支持好
    - 部署简单，适合个人和小团队使用
    - 成本控制好，无需购买额外服务
  - 代价/风险：
    - SQLite 并发性能有限，不适合高并发场景
    - OpenClaw 依赖浏览器环境，可能存在兼容性问题
- 备选方案：
  - Node.js + MongoDB + Puppeteer —— 技术栈较新但生态不如 Python 成熟
  - Java + PostgreSQL + Selenium —— 企业级但开发复杂度过高
- 参考：
  - `/src/config.py` - 配置管理实现
  - `/src/database.py` - 数据库模型设计

## ADR-002：AI 模型选择策略 - GLM-5 + OpenRouter
- 日期：2026-02-25
- 状态：Accepted（已采纳）
- 背景（要解决的问题）：
  - 需要平衡内容质量、生成速度和成本控制，实现智能内容创作
- 决策：
  - 主力使用 GLM-5 系列模型（GLM-5-Flash、GLM-5）
  - 高级任务使用 OpenRouter 的 Claude Sonnet 4.6 / Opus 4.6
  - 轻量任务使用 GLM-5-Flash / GPT-4o-mini
- 原因（为什么）：
  - GLM-5 性价比高，中文处理能力强，适合日常内容生成
  - Claude 模型在创意和推理方面表现优秀，适合复杂任务
  - 多模型策略可以平衡质量、速度和成本
  - OpenRouter 提供统一 API 接口，简化集成复杂度
- 影响（后果）：
  - 正向：
    - 每日成本控制在 ¥20 以内
    - 内容质量有保障，支持多种任务类型
    - 故障时可以降级使用低成本模型
  - 代价/风险：
    - 多 API 集成增加代码复杂度
    - 依赖外部服务稳定性
- 备选方案：
  - 单一模型（如只使用 GPT-4）—— 成本过高，缺乏灵活性
  - 纯本地模型（如 Llama）—— 质量不稳定，部署复杂
- 参考：
  - `/src/ai/engine.py` - AI 引擎实现
  - `/src/ai/models.py` - 模型选择策略

## ADR-003：架构设计 - 模块化六阶段流程
- 日期：2026-02-25
- 状态：Accepted（已采纳）
- 背景（要解决的问题）：
  - 需要设计可扩展、可维护的自动化运营系统架构
- 决策：
  - 采用模块化六阶段流程：素材搜索 → 内容生成 → 自动发布 → 主动互动 → 数据分析 → 复盘调整
  - 每个模块独立开发，通过数据流串联
  - 支持单模块测试和调试
- 原因（为什么）：
  - 模块化设计便于开发和维护
  - 数据流清晰，便于追踪和调试
  - 支持渐进式开发，降低风险
  - 便于扩展新功能模块
- 影响（后果）：
  - 正向：
    - 开发风险分散，问题定位容易
    - 代码复用性高，测试覆盖容易
    - 便于团队协作开发
  - 代价/风险：
    - 模块间依赖需要管理
    - 数据传递和错误处理复杂
- 备选方案：
  - 单体架构 —— 开发简单但维护困难
  - 微服务架构 —— 过度设计，增加部署复杂度
- 参考：
  - `/src/main.py` - 主流程调度
  - `/src/scheduler/` - 调度器实现

## ADR-004：成本控制策略 - 预算限制和模型选择
- 日期：2026-02-25
- 状态：Accepted（已采纳）
- 背景（要解决的问题）：
  - 需要将每日运营成本控制在合理范围内，确保项目可持续性
- 决策：
  - 设定每日预算上限 ¥20
  - 每 4 小时一个循环，每轮成本约 ¥1.2
  - 根据任务复杂度自动选择成本最优模型
- 原因（为什么）：
  - ¥20/日预算合理，可以支持高质量内容生成
  - 4 小时间隔平衡了内容频率和成本控制
  - 智能模型选择最大化性价比
  - 预算限制避免意外成本超支
- 影响（后果）：
  - 正向：
    - 成本可控，项目可持续
    - 避免意外高额账单
    - 促进优化算法和策略
  - 代价/风险：
    - 可能需要牺牲部分内容质量
    - 复杂任务可能受限
- 备选方案：
  - 无预算限制 —— 成本不可控，风险高
  - 更低预算（如 ¥10/日）—— 内容质量可能受影响
- 参考：
  - `/src/utils/cost_tracker.py` - 成本追踪实现
  - `/src/config.py` - 预算配置

## ADR-005：部署方案 - 本地优先 + 云服务器备选
- 日期：2026-02-25
- 状态：Accepted（已采纳）
- 背景（要解决的问题）：
  - 需要简单可靠的部署方案，支持开发和测试环境
- 决策：
  - 优先支持本地部署和运行
  - 提供云服务器部署脚本和 systemd 服务配置
  - 支持 Docker 容器化部署（可选）
- 原因（为什么）：
  - 本地部署便于开发和调试
  - 云服务器部署保证 24/7 稳定运行
  - Docker 提供环境一致性
  - 多部署方案满足不同需求
- 影响（后果）：
  - 正向：
    - 开发调试方便
    - 生产环境稳定
    - 部署灵活性强
  - 代价/风险：
    - 需要维护多套部署配置
    - 环境差异可能导致问题
- 备选方案：
  - 纯云部署 —— 开发调试困难
  - 纯本地部署 —— 无法保证 24/7 运行
- 参考：
  - `/scripts/` - 部署脚本
  - `/scripts/systemd/xplan.service` - systemd 服务配置

## ADR-006：导航栏重构 - 二级下拉菜单
- 日期：2026-03-24
- 状态：Accepted（已采纳）
- 背景（要解决的问题）：
  - Phase 3 新增储蓄计划、宝宝基金、批量导入后，一级导航达到 7 项，层级混乱
  - 高频功能（首页、报表）和低频功能（分类管理、批量导入）平铺在同一层级
- 决策：
  - 重构为 4 个一级项 + 2 个下拉菜单
  - 首页 | 账户 ▾ (账户管理/储蓄计划/宝宝基金) | 报表 | 设置 ▾ (分类管理/批量导入)
  - CSS hover 触发下拉菜单，无需 JavaScript
- 原因（为什么）：
  - 按功能域分组更符合用户心智模型
  - 一级导航精简后视觉更清爽
  - hover 触发比点击触发更轻量，减少操作步骤
- 影响（后果）：
  - 正向：导航层级清晰，一级只有 4 项
  - 代价：常用的账户管理需要 hover 后再点击（多一步）
- 备选方案：
  - 方案 A：三组（资产/报表/工具）+ 宝宝基金独立 —— 一级 5 项略多
  - 全部平铺 —— 已被否决，7 项过多
