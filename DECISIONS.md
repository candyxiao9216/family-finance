## ADR-001：技术栈选择 - Python + SQLite + OpenClaw
- 日期：2026-02-25
- 状态：Accepted（已采纳）
- 背景（要解决的问题）：
  - 需要快速开发全自动 Twitter 运营系统，平衡开发效率、成本和功能需求
- 决策：
  - 使用 Python 3.8+ 作为主要开发语言
  - 使用 SQLite 作为本地数据存储
  - 使用 OpenClaw 进行浏览器自动化控制
- 原因（为什么）：
  - Python 生态系统丰富，AI 集成和数据处理能力强
  - SQLite 轻量级，适合单机部署，无需额外数据库服务
  - OpenClaw 提供稳定的浏览器控制能力，避免反爬虫检测
  - 技术栈组合简单，部署和维护成本低
- 影响（后果）：
  - 正向：
    - 开发速度快，社区支持好
    - 部署简单，适合个人和小团队使用
    - 成本控制好，无需购买额外服务
  - 代价/风险：
    - SQLite 并发性能有限，不适合高并发场景
    - OpenClaw 依赖浏览器环境，可能存在兼容性问题
- 备选方案：
  - Node.js + MongoDB + Puppeteer —— 技术栈较新但生态不如 Python 成熟
  - Java + PostgreSQL + Selenium —— 企业级但开发复杂度过高
- 参考：
  - `/src/config.py` - 配置管理实现
  - `/src/database.py` - 数据库模型设计

## ADR-002：AI 模型选择策略 - GLM-5 + OpenRouter
- 日期：2026-02-25
- 状态：Accepted（已采纳）
- 背景（要解决的问题）：
  - 需要平衡内容质量、生成速度和成本控制，实现智能内容创作
- 决策：
  - 主力使用 GLM-5 系列模型（GLM-5-Flash、GLM-5）
  - 高级任务使用 OpenRouter 的 Claude Sonnet 4.6 / Opus 4.6
  - 轻量任务使用 GLM-5-Flash / GPT-4o-mini
- 原因（为什么）：
  - GLM-5 性价比高，中文处理能力强，适合日常内容生成
  - Claude 模型在创意和推理方面表现优秀，适合复杂任务
  - 多模型策略可以平衡质量、速度和成本
  - OpenRouter 提供统一 API 接口，简化集成复杂度
- 影响（后果）：
  - 正向：
    - 每日成本控制在 ¥20 以内
    - 内容质量有保障，支持多种任务类型
    - 故障时可以降级使用低成本模型
  - 代价/风险：
    - 多 API 集成增加代码复杂度
    - 依赖外部服务稳定性
- 备选方案：
  - 单一模型（如只使用 GPT-4）—— 成本过高，缺乏灵活性
  - 纯本地模型（如 Llama）—— 质量不稳定，部署复杂
- 参考：
  - `/src/ai/engine.py` - AI 引擎实现
  - `/src/ai/models.py` - 模型选择策略

## ADR-003：架构设计 - 模块化六阶段流程
- 日期：2026-02-25
- 状态：Accepted（已采纳）
- 背景（要解决的问题）：
  - 需要设计可扩展、可维护的自动化运营系统架构
- 决策：
  - 采用模块化六阶段流程：素材搜索 → 内容生成 → 自动发布 → 主动互动 → 数据分析 → 复盘调整
  - 每个模块独立开发，通过数据流串联
  - 支持单模块测试和调试
- 原因（为什么）：
  - 模块化设计便于开发和维护
  - 数据流清晰，便于追踪和调试
  - 支持渐进式开发，降低风险
  - 便于扩展新功能模块
- 影响（后果）：
  - 正向：
    - 开发风险分散，问题定位容易
    - 代码复用性高，测试覆盖容易
    - 便于团队协作开发
  - 代价/风险：
    - 模块间依赖需要管理
    - 数据传递和错误处理复杂
- 备选方案：
  - 单体架构 —— 开发简单但维护困难
  - 微服务架构 —— 过度设计，增加部署复杂度
- 参考：
  - `/src/main.py` - 主流程调度
  - `/src/scheduler/` - 调度器实现

## ADR-004：成本控制策略 - 预算限制和模型选择
- 日期：2026-02-25
- 状态：Accepted（已采纳）
- 背景（要解决的问题）：
  - 需要将每日运营成本控制在合理范围内，确保项目可持续性
- 决策：
  - 设定每日预算上限 ¥20
  - 每 4 小时一个循环，每轮成本约 ¥1.2
  - 根据任务复杂度自动选择成本最优模型
- 原因（为什么）：
  - ¥20/日预算合理，可以支持高质量内容生成
  - 4 小时间隔平衡了内容频率和成本控制
  - 智能模型选择最大化性价比
  - 预算限制避免意外成本超支
- 影响（后果）：
  - 正向：
    - 成本可控，项目可持续
    - 避免意外高额账单
    - 促进优化算法和策略
  - 代价/风险：
    - 可能需要牺牲部分内容质量
    - 复杂任务可能受限
- 备选方案：
  - 无预算限制 —— 成本不可控，风险高
  - 更低预算（如 ¥10/日）—— 内容质量可能受影响
- 参考：
  - `/src/utils/cost_tracker.py` - 成本追踪实现
  - `/src/config.py` - 预算配置

## ADR-005：部署方案 - 本地优先 + 云服务器备选

- 背景（要解决的问题）：

    - 需要维护多套部署配置
    - 环境差异可能导致问题
- 备选方案：
  - 纯云部署 —— 开发调试困难
  - 纯本地部署 —— 无法保证 24/7 运行
- 参考：

## ADR-006：导航栏重构 - 二级下拉菜单
- 日期：2026-03-24
- 状态：Accepted（已采纳）
- 背景（要解决的问题）：
  - Phase 3 新增储蓄计划、宝宝基金、批量导入后，一级导航达到 7 项，层级混乱
  - 高频功能（首页、报表）和低频功能（分类管理、批量导入）平铺在同一层级
- 决策：
  - 重构为 4 个一级项 + 2 个下拉菜单
  - 首页 | 账户 ▾ (账户管理/储蓄计划/宝宝基金) | 报表 | 设置 ▾ (分类管理/批量导入)
  - CSS hover 触发下拉菜单，无需 JavaScript
- 原因（为什么）：
  - 按功能域分组更符合用户心智模型
  - 一级导航精简后视觉更清爽
  - hover 触发比点击触发更轻量，减少操作步骤
- 影响（后果）：
  - 正向：导航层级清晰，一级只有 4 项
  - 代价：常用的账户管理需要 hover 后再点击（多一步）
- 备选方案：
  - 方案 A：三组（资产/报表/工具）+ 宝宝基金独立 —— 一级 5 项略多
  - 全部平铺 —— 已被否决，7 项过多

## ADR-007：移动端导航 - 混合方案（底部 Tab + 汉堡菜单）
- 日期：2026-03-26
- 状态：Accepted（已采纳）
- 背景（要解决的问题）：
  - 桌面端二级下拉菜单依赖鼠标 hover，移动端无法使用
  - 需要选择适合手机的导航方式
- 决策：
  - 移动端（<768px）采用混合方案：底部 Tab 栏 + 顶部汉堡侧滑菜单
  - 底部 Tab：4 个高频入口（首页/账户/报表/更多）
  - 汉堡菜单：点击 ☰ 或「更多」Tab 时右侧滑出全屏菜单
  - 桌面端（≥768px）保持原有二级下拉导航不变
- 原因（为什么）：
  - 底部 Tab 让高频功能始终可达（一触即达）
  - 汉堡菜单容纳所有低频功能，不占用主界面空间
  - 混合方案兼顾了信息密度和操作便捷
- 影响（后果）：
  - 正向：手机用户体验大幅提升，高频操作减少到 1 次点击
  - 代价：实现复杂度高于单一方案（需 JS 控制菜单状态）
- 备选方案：
  - 方案 A：纯汉堡菜单 —— 简单但所有功能都藏在菜单里，多一步操作
  - 方案 B：纯底部 Tab —— 位置有限，二级页面需要跳转

## ADR-008：模板重构 - base.html 继承体系
- 日期：2026-03-26
- 状态：Accepted（已采纳）
- 背景（要解决的问题）：
  - 12 个模板各自重复约 40 行 head+nav 代码
  - 每次修改导航需要改 12 个文件，容易遗漏
- 决策：
  - 抽取 `base.html`（含导航、Tab、Toast、确认弹窗）和 `auth_base.html`（仅 head）
  - 10 个页面继承 `base.html`，2 个认证页继承 `auth_base.html`
  - 提供 5 个 block：title、extra_head、mobile_title、content、scripts
- 原因（为什么）：
  - 消除重复代码（减少约 450 行）
  - 导航/Toast/确认弹窗等公共组件只维护一处
  - Jinja2 模板继承是 Flask 标准实践
- 影响（后果）：
  - 正向：维护成本大幅降低，新增页面只需 extends + block
  - 代价：一次性改动 12 个模板，改造成本较高

## ADR-009：首页重做 - 从记账页改为仪表盘
- 日期：2026-04-05
- 状态：Accepted（已采纳）
- 背景（要解决的问题）：
  - 原首页是纯记账页（统计栏+表单+交易列表），用户无法一眼看到全局财务状况
  - 资产总览和储蓄进度需要分别进入子页面才能查看
- 决策：
  - 首页改为三模块概览仪表盘（月度收支+资产总览+储蓄计划）
  - 原记账功能迁移到独立的「月度收支」页面（`/transactions`）
  - 导航更新：账户→资产总览，储蓄→储蓄计划，管理▾增加月度收支
- 原因（为什么）：
  - 仪表盘让用户登录后一眼掌握全局财务状况
  - 三大核心指标（收支/资产/储蓄）在一个页面汇总，减少跳转
  - 记账功能迁移到独立页面后更聚焦，不干扰概览体验
- 影响（后果）：
  - 正向：用户体验提升，首页信息密度更高
  - 代价：记账操作多一步（从首页点击「查看详情」进入月度收支页）

## ADR-010：小眼睛金额隐藏 - 全局统一默认隐藏
- 日期：2026-04-05
- 状态：Accepted（已采纳）
- 背景（要解决的问题）：
  - 原小眼睛只覆盖 `.stat-value`，新仪表盘使用不同 class 无法被隐藏
  - 需要所有页面默认隐藏金额，保护隐私
- 决策：
  - 扩展选择器覆盖所有金额 class：`.stat-value, .dash-stat-value, .asset-row-value, .amount-hide`
  - 所有页面默认隐藏（localStorage 无值时 isHidden=true）
  - 暴露 `window.ffReapplyHide()` 解决 JS 动态赋值覆盖问题
  - 进度条和百分比不隐藏（只隐藏金额数字）
- 原因（为什么）：
  - 财务数据敏感，公共场合默认隐藏更安全
  - 统一选择器避免新页面遗漏
  - ffReapplyHide 机制解决了页面 JS 晚于 app.js 执行的时序问题

## ADR-011：AI 模型选择 - 智谱GLM全系列
- 日期：2026-04-07
- 状态：Accepted（已采纳）
- 背景：财务顾问模块需要 AI 分析能力，同时为后续图片OCR导入做准备
- 决策：
  - 文本分析：GLM-5（旗舰模型，财务建议质量优先）
  - 多模态理解：GLM-5V-Turbo（截图OCR，持仓数据提取）
  - 图像生成：GLM-Image（预留能力）
  - 统一使用智谱开放平台，一个 API Key 三种模型
- 原因：
  - 智谱 API 兼容 OpenAI 格式，集成成本低
  - GLM-5 中文理解能力强，适合财务场景
  - 一个平台覆盖文本+视觉+图像，减少多供应商管理
- 备选方案：MiniMax（原方案）、OpenRouter（多模型路由）

## ADR-012：AI 分析展示 - 右侧抽屉交互
- 日期：2026-04-07
- 状态：Accepted（已采纳）
- 背景：AI 分析结果需要一个展示区域，原底部面板空间不足且遮挡内容
- 决策：
  - 使用右侧抽屉（400px宽，移动端全屏）展示 AI 分析结果
  - 全局组件 `window.aiDrawer`，所有页面共享
  - 支持：Markdown渲染、时间戳、缓存标记、强制刷新、历史入口
- 原因：
  - 抽屉不遮挡主内容，可同时查看持仓数据和 AI 建议
  - 全局组件复用，5个页面无需各自实现
  - Markdown 渲染让 AI 输出格式更丰富（表格、列表、加粗）
- 备选方案：底部折叠面板（空间小）、模态框（全遮挡）、页内嵌入（占固定空间）
