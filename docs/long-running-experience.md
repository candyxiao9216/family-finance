# 长时间自主运行经验总结

## 为什么这次能跑 1 小时无人干预

### 核心原因：充分的前期讨论 + 明确的实施计划

这次任务在写代码之前，花了大量时间做需求讨论和方案设计。等到真正开始执行时，每一步该做什么、怎么做、验收标准是什么，都已经清晰定义好了。

---

## 可复用的经验

### 1. 需求讨论阶段：一次只问一个问题

这次讨论了 6 个设计问题，每次只抛一个问题 + 2-3 个选项 + 推荐理由。这样做的好处：
- 用户不会被一堆问题淹没
- 每个决策都有明确记录
- 不会出现"我以为你说的是..."的歧义

**模板：**
```
问题：XXX 怎么实现？
方案 A：...（优点/缺点）
方案 B：...（优点/缺点）
我建议方案 X，因为...
```

### 2. 写设计文档再写代码

讨论完所有问题后，输出了一份完整的设计文档（`docs/plans/2026-02-27-family-sharing-design.md`），包含：
- 数据库表结构（精确到字段和类型）
- API 路由列表
- 核心业务流程
- 验收标准

这份文档就是 AI 执行任务的"说明书"，有了它就不需要反复问你。

### 3. 实施计划拆成小步骤

设计文档之后，又写了一份实施计划，把大任务拆成 12 个小任务，每个任务包含：
- 要改哪些文件（精确到行号）
- 具体的代码片段
- 测试命令和预期输出
- 提交信息

**关键点：每个任务足够小（5-15 分钟），独立可验证。**

### 4. 使用 Subagent 隔离上下文

每个小任务分发给一个独立的子代理（subagent）执行，好处：
- 子代理只关注当前任务，不会被之前的上下文干扰
- 一个任务失败不会污染其他任务
- 主代理负责协调和审查，保持全局视野

### 5. 两阶段审查机制

每个任务完成后做两轮审查：
1. **规范符合性审查**：代码是否满足设计文档的要求
2. **代码质量审查**：代码是否健壮、可维护

这确保了质量，不需要人工介入检查。

## 你下次可以直接用的 Prompt 模板

```
我要做 XXX 功能，在我认可需求之前不要开始写代码。

请：
1. 先了解当前项目状态
2. 逐个问我设计问题（每次一个，给选项）
3. 所有问题确认后，写设计文档
4. 写实施计划（拆成小任务）
5. 用 subagent 逐个执行任务
```

---

## 总结：三个关键条件

| 条件 | 说明 |
|------|------|
| **需求无歧义** | 前期充分讨论，所有决策都记录在案 |
| **计划够细致** | 每个任务有明确的输入、输出、验收标准 |
| **执行可验证** | 每步完成后有审查机制，不依赖人工判断 |

满足这三个条件，AI 就能长时间自主运行而不偏离方向。
